# Высокоуровневое техническое решение (HLD)

## 1. Цель документа

Определить технический подход к реализации поддержки нескольких городов в скрипте `/home/hermes_ai/.hermes/scripts/weather_daily.py` на основе BRD.

## 2. Контекст

- **AS-IS:** Скрипт жёстко привязан к Москве (LAT, LON, CITY, TIMEZONE — константы).
- **TO-BE:** Пользователь передаёт список городов; скрипт геокодирует каждый город, запрашивает прогноз из Open-Meteo и формирует отдельное Telegram-сообщение в текущем формате.
- **Город по умолчанию:** Москва, если аргументы не указаны.

## 3. Архитектура

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  CLI args / env │────▶│  weather_daily   │────▶│ Geocoding API   │
│  (список городов)│     │  (Python 3.11)   │     │ Open-Meteo      │
└─────────────────┘     └──────────────────┘     └─────────────────┘
                              │    │                       │
                              │    ▼                       ▼
                              │ ┌─────────────────┐   ┌─────────────────┐
                              │ │ Локальный кэш   │   │ Weather API     │
                              │ │ координат (JSON)│   │ Open-Meteo      │
                              │ └─────────────────┘   └─────────────────┘
                              ▼
                        ┌─────────────────┐
                        │ Печать сообщений│
                        │ (Telegram/Hermes)│
                        └─────────────────┘
```

## 4. Передача списка городов

**Решение:** поддержать три источника входных данных в порядке приоритета:

1. **Аргументы командной строки** — основной способ. Например:
   ```bash
   python weather_daily.py Москва "Санкт-Петербург" Казань
   ```
2. **Переменная окружения `WEATHER_CITIES`** — для cron / автоматизации:
   ```bash
   WEATHER_CITIES="Москва, СПб, Казань" python weather_daily.py
   ```
3. **Файл конфигурации `~/.config/weather_daily/cities.json`** — список городов по умолчанию для профиля пользователя.

**Приоритет:** CLI args → env → config → default (Москва).

## 5. Геокодирование

**Решение:** использовать **Open-Meteo Geocoding API** (`https://geocoding-api.open-meteo.com/v1/search`).

- Бесплатный, без ключа, совместим с Open-Meteo Weather API.
- URL: `https://geocoding-api.open-meteo.com/v1/search?name=<city>&count=1&language=ru&format=json`.
- Берём первый результат (`results[0]`) — соответствует BRULE-05.
- Из ответа извлекаем: `latitude`, `longitude`, `name`, `timezone` (если есть; иначе `timezone=auto` в Open-Meteo).

**Обработка ошибок:**
- HTTP-ошибка / timeout → логируем, город пропускается, скрипт продолжает.
- Пустой результат (`results` отсутствует) → сообщение: `❌ Город "X" не найден`.
- Для сохранения NFR-03: код возврата ≠ 0, если хотя бы один город завершился ошибкой.

## 6. Кэширование координат

**Решение:** ввести **локальный JSON-кэш** координат.

- Путь: `~/.cache/weather_daily/geocache.json`.
- Структура:
  ```json
  {
    "москва": {
      "lat": 55.7558,
      "lon": 37.6173,
      "name": "Москва",
      "timezone": "Europe/Moscow",
      "cached_at": "2026-07-18T12:00:00"
    }
  }
  ```
- Ключ кэша — нормализованное название города (нижний регистр, обрезаны пробелы).
- TTL: **30 дней** (конфигурируемый параметр).
- Применение: перед вызовом геокодера проверяем кэш; при промахе — делаем запрос и сохраняем результат.
- Кэш снижает потребление внешнего API (NFR-04) и закрывает риск R-01.

## 7. Обработка часовых поясов

**Решение:**
- Если геокодер вернул `timezone` — используем его.
- Иначе передаём `timezone=auto` в Open-Meteo Forecast API.
- Это обеспечивает корректные периоды суток для иностранных городов без подключения дополнительных библиотек (`pytz` не требуется).

## 8. Формирование сообщений

**Решение:**
- Для каждого города формируется **отдельное сообщение** (BR-03).
- Используем существующий формат с заголовком `🌍 *{city}* — прогноз на сегодня`.
- Название города в заголовке берётся из геокодера (`name`) — соответствует BR-06.
- Сохраняем текущий `build_message(data, city, timezone)`.

## 9. Ошибки и логирование

**Решение:**
- Вести `logging` вместо `print` для диагностики:
  - `INFO`: обработка города, кэш-хиты/промахи.
  - `WARNING`: город не найден, некорректный ввод.
  - `ERROR`: сетевые ошибки, падение по городу.
- Пользовательский вывод (Telegram) остаётся через `print` / `send_telegram`.
- При ошибке по отдельному городе выводится короткое сообщение и обработка продолжается (BR-05, NFR-03).

## 10. Backward compatibility

**Решение:**
- Запуск без аргументов возвращает прогноз для Москвы (BR-02).
- Старые константы `LAT`, `LON`, `CITY`, `TIMEZONE` сохраняются как значения по умолчанию (fallback).
- Внешний интерфейс (вывод в stdout) не изменяется за исключением добавления городов.

## 11. Структура модуля (предлагаемые изменения)

```text
weather_daily.py
├── parse_city_args()         # CLI / env / config
├── Geocache                  # класс для работы с ~/.cache/weather_daily/geocache.json
├── geocode_city(name)        # кэш → Open-Meteo Geocoding API

├── build_message(data, city) # существующий формат
└── main()                    # цикл по городам, обработка ошибок, exit code
```

## 12. Зависимости

**Решение:** не добавлять новые сторонние Python-пакеты. Используем только стандартную библиотеку:
- `argparse`
- `json`
- `os`
- `pathlib`
- `urllib.request`
- `datetime`
- `logging`

Это упрощает развёртывание в cron-заданиях.

## 13. Границы и ограничения

- Максимум 10 городов за запуск (BRULE-02) — проверяем в `parse_city_args()`.
- Поддержка русских и латинских названий — через URL-кодирование и `language=ru` в геокодере.
- Каждый город обрабатывается последовательно (BRULE-03); параллельность вынесена за рамки текущей задачи.

## 14. Открытые вопросы, требующие решения бизнес-заказчика

| # | Вопрос | Рекомендуемое решение |
|---|--------|------------------------|
| 1 | Лимит в 10 городов | Оставить 10; регулируется константой `MAX_CITIES`. |
| 2 | Выбор геокодера | Open-Meteo Geocoding API — бесплатно, без ключа, без CORS-проблем. |
| 3 | Кэширование координат | Включить локальный JSON-кэш с TTL 30 дней. |
| 4 | Иностранные города и латиница | Поддержать; часовой пояс через `timezone=auto` в Open-Meteo. |

## 15. Критерии приёмки реализации

- `python weather_daily.py` → прогноз для Москвы.
- `python weather_daily.py Москва "Санкт-Петербург"` → два отдельных сообщения.
- `python weather_daily.py НесуществующийГород Москва` → ошибка по первому городу + прогноз для Москвы, exit code ≠ 0.
- 11 городов → ошибка с пояснением лимита.
- Повторный запуск того же города использует кэш (нет лишнего HTTP-вызова).
