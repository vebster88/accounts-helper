### ТК-05: Конфигурационный файл

**Команда:** `python weather_daily.py` при наличии `~/.config/weather_daily/cities.json`

**Код возврата:** `0` (но использован город по умолчанию)

**Дефект:**
- Спецификация в разделе 4.3 и примере JSON-схемы (раздел 5.2) описывает файл как **массив городов** `["Владивосток", "Новосибирск"]`.
- Реализация в `parse_city_sources` пытается обработать `list` через вложенный генератор с ошибкой:
  ```python
  return [city for city in (_split_city_list(item) if isinstance(item, str) else [item])
          for city in (city if isinstance(city, list) else [city]) if isinstance(city, str) and city.strip()]
  ```
  Здесь переменная `item` не определена в текущей области, что приводит к исключению:
  ```
  name 'item' is not defined
  ```
- В результате конфигурация игнорируется, и используется Москва по умолчанию.

**Соответствие ТЗ:** FR-04 — **не выполнено для формата массива**. Требуется исправление парсинга конфигурационного файла.

## Топ-3 находки

### 1. [ДЕФЕКТ] Конфигурационный файл не читается

- **Код ТЗ:** FR-04
- **Серьёзность:** Medium
- **Описание:** При использовании корректного JSON-формата `["Владивосток", "Носибирск"]` скрипт падает в `parse_city_sources` с исключением `name 'item' is not defined`. Конфиг игнорируется, подставляется Москва по умолчанию.
- **Рекомендация:** Упростить парсинг конфига: поддерживать строку и список строк, убрать ошибочный вложенный генератор.

### 2. [ДЕФЕКТ] Формат Telegram-сообщения отличается от ТЗ

- **Код ТЗ:** раздел 8.1
- **Серьёзность:** Low
- **Описание:** ТЗ требует формат:
  ```
  🌡 Сейчас: +{current_temp}°C, {current_condition}
  🌅 Утро (06-12): +{morning_temp}°C, {morning_condition}
  ```
  Скрипт выводит расширенный формат с дополнительными полями (ощущаемая температура, влажность, ветер, разделители `───`, источник данных и дата). При этом базовые требования (температура, WMO-описание, периоды суток) присутствуют.
- **Рекомендация:** При необходимости строгого соответствия ТЗ привести формат сообщения к шаблону из раздела 8.1.

### 3. [НАБЛЮДЕНИЕ] Период «Ночь» для иностранного города может быть без данных

- **Код ТЗ:** FR-07, раздел 7.5
- **Серьёзность:** Low
- **Описание:** Для Лондона в выводе `🌙 Ночь: нет данных`. Это связано с тем, что Open-Meteo Forecast API при `forecast_days=2` возвращает почасовые данные, но для удалённого часового пояса часть ночных часов следующего дня может отсутствовать в текущем срезе ответа. Скрипт корректно обрабатывает отсутствие данных, однако это может влиять на полноту прогноза для городов с сильным смещением пояса.
- **Рекомендация:** Убедиться, что `forecast_days` достаточно для покрытия всех периодов суток в местном времени, либо документировать данное поведение.
