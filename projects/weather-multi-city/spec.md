## 4. Аргументы командной строки

### 4.1. Позиционные аргументы

```text
python weather_daily.py [ГОРОД ...]
```

- Каждый город — отдельный позиционный аргумент.
- Названия из нескольких слов передаются в кавычках.
- Регистр не влияет на поиск; исходное написание может использоваться для отображения при отсутствии результата геокодера.

### 4.2. Примеры запуска

| Сценарий | Команда | Ожидаемый результат |
|---|---|---|
| Один город по умолчанию | `python weather_daily.py` | Прогноз для Москвы |
| Один город | `python weather_daily.py Санкт-Петербург` | Прогноз для Санкт-Петербурга |
| Несколько городов | `python weather_daily.py Москва "Санкт-Петербург" Казань` | Три отдельных сообщения |
| Латиница | `python weather_daily.py Moscow London` | Два сообщения на английских названиях |
| Ошибка + успех | `python weather_daily.py НесуществующийГород Москва` | Ошибка по первому городу + прогноз для Москвы, `exit code ≠ 0` |
| Превышение лимита | `python weather_daily.py г1 г2 ... г11` | Ошибка «максимум 10 городов», `exit code = 2` |

### 4.3. Приоритет источников списка городов

1. Позиционные аргументы CLI.
2. Переменная окружения `WEATHER_CITIES`.
3. Файл `~/.config/weather_daily/cities.json`.
4. Значение по умолчанию — `Москва`.

Переменная окружения и файл конфигурации интерпретируются как список, разделённый запятыми. Пробелы вокруг запятых игнорируются.

## 5. JSON-схема файла кэша

### 5.1. Путь к файлу

```text
~/.cache/weather_daily/geocache.json
```

### 5.2. Содержимое

```json
{
  "москва": {
    "lat": 55.7558,
    "lon": 37.6173,
    "name": "Москва",
    "timezone": "Europe/Moscow",
    "cached_at": "2026-07-18T12:00:00"
  },
  "санкт-петербург": {
    "lat": 59.9386,
    "lon": 30.3141,
    "name": "Санкт-Петербург",
    "timezone": "Europe/Moscow",
    "cached_at": "2026-07-18T12:00:00"
  }
}
```

### 5.3. Поля записи

| Поле | Тип | Обязательность | Описание |
|---|---|---|---|
| `lat` | number | Да | Широта города |
| `lon` | number | Да | Долгота города |
| `name` | string | Да | Официальное название города из геокодера |
| `timezone` | string | Нет | Часовой пояс; если отсутствует, используется `timezone=auto` |
| `cached_at` | string (ISO 8601) | Да | Дата и время сохранения записи |

### 5.4. Правила работы с кэшем

- Ключ — нормализованное название: нижний регистр, удалены лишние пробелы.
- Перед геокодированием проверяется кэш.
- Если запись есть и не старше 30 дней — используется кэш.
- Если записи нет или TTL истёк — выполняется запрос к геокодеру, результат сохраняется в кэш.
- Файл кэша создаётся автоматически при первом успешном геокодировании.

## 6. Контракт Open-Meteo Geocoding API

### 6.1. Endpoint

```text
GET https://geocoding-api.open-meteo.com/v1/search
```

### 6.2. Параметры запроса

| Параметр | Тип | Обязательность | Значение | Описание |
|---|---|---|---|---|
| `name` | string | Да | URL-encoded название города | Название города для поиска |
| `count` | integer | Нет | `1` | Количество возвращаемых результатов |
| `language` | string | Нет | `ru` | Предпочтительный язык ответа |
| `format` | string | Нет | `json` | Формат ответа |

### 6.3. Пример запроса

```text
https://geocoding-api.open-meteo.com/v1/search?name=%D0%9C%D0%BE%D1%81%D0%BA%D0%B2%D0%B0&count=1&language=ru&format=json
```

### 6.4. Пример ответа (успех)

```json
{
  "results": [
    {
      "id": 524901,
      "name": "Москва",
      "latitude": 55.7558,
      "longitude": 37.6173,
      "elevation": 144.0,
      "feature_code": "PPLC",
      "country_code": "RU",
      "admin1_id": 524894,
      "admin2_id": 567380,
      "timezone": "Europe/Moscow",
      "population": 10381222,
      "country": "Россия",
      "admin1": "Москва"
    }
  ],
  "generationtime_ms": 0.123
}
```

### 6.5. Используемые поля ответа

| Поле | Тип | Назначение |
|---|---|---|
| `results` | array | Список найденных городов |
| `results[0].name` | string | Отображаемое название города |
| `results[0].latitude` | number | Широта |
| `results[0].longitude` | number | Долгота |
| `results[0].timezone` | string | Часовой пояс города |

### 6.6. Поведение при отсутствии результатов

- Если поле `results` отсутствует или пустое — город считается не найденным.
- Выводится пользовательское сообщение об ошибке по городу.

## 7. Контракт Open-Meteo Forecast API (для нескольких координат)

### 7.1. Endpoint

```text
GET https://api.open-meteo.com/v1/forecast
```

### 7.2. Параметры запроса

| Параметр | Тип | Обязательность | Значение | Описание |
|---|---|---|---|---|
| `latitude` | number | Да | Широта города | Координата |
| `longitude` | number | Да | Долгота города | Координата |
| `timezone` | string | Да | `auto` или значение из геокодера | Часовой пояс |
| `current` | string | Да | `temperature_2m,weather_code` | Текущие параметры |
| `hourly` | string | Да | `temperature_2m,weather_code` | Почасовые параметры |
| `forecast_days` | integer | Нет | `1` | Количество дней прогноза |

### 7.3. Пример запроса

```text
https://api.open-meteo.com/v1/forecast?latitude=55.7558&longitude=37.6173&timezone=Europe%2FMoscow&current=temperature_2m,weather_code&hourly=temperature_2m,weather_code&forecast_days=1
```

### 7.4. Используемые поля ответа

| Поле | Тип | Назначение |
|---|---|---|
| `current.temperature_2m` | number | Текущая температура |
| `current.weather_code` | integer | WMO-код текущей погоды |
| `hourly.time` | array[string] | Метки времени в формате ISO 8601 |
| `hourly.temperature_2m` | array[number] | Температура по часам |
| `hourly.weather_code` | array[integer] | WMO-коды по часам |

### 7.5. Агрегация по периодам суток

| Период | Часы (местное время) | Способ агрегации |
|---|---|---|
| Утро | 06–12 | Средняя температура; доминирующий WMO-код |
| День | 12–18 | Средняя температура; доминирующий WMO-код |
| Вечер | 18–24 | Средняя температура; доминирующий WMO-код |
| Ночь | 00–06 | Средняя температура; доминирующий WMO-код |

## 8. Формат Telegram-сообщения

### 8.1. Один город

```text
🌍 *{city}* — прогноз на сегодня

🌡 Сейчас: +{current_temp}°C, {current_condition}

🌅 Утро (06-12): +{morning_temp}°C, {morning_condition}
☀️ День (12-18): +{day_temp}°C, {day_condition}
🌆 Вечер (18-24): +{evening_temp}°C, {evening_condition}
🌙 Ночь (00-06): +{night_temp}°C, {night_condition}
```

### 8.2. Несколько городов

Для каждого города формируется отдельное сообщение в формате 8.1. Сообщения разделяются пустой строкой или выводятся последовательно — в зависимости от окружения печати.

Пример для двух городов:

```text
🌍 *Москва* — прогноз на сегодня

🌡 Сейчас: +18°C, Переменная облачность

🌅 Утро (06-12): +16°C, Облачно
☀️ День (12-18): +22°C, Ясно
🌆 Вечер (18-24): +19°C, Переменная облачность
🌙 Ночь (00-06): +14°C, Ясно

🌍 *Санкт-Петербург* — прогноз на сегодня

🌡 Сейчас: +15°C, Пасмурно

🌅 Утро (06-12): +14°C, Пасмурно
☀️ День (12-18): +17°C, Облачно
🌆 Вечер (18-24): +16°C, Облачно
🌙 Ночь (00-06): +13°C, Пасмурно
```

### 8.3. Замечания по формату

- Заголовок содержит название города из результата геокодирования.
- Знак температуры выводится явно (`+` / `-`).
- WMO-код преобразуется в читаемое описание погоды.
- При ошибке по городу выводится короткое сообщение:
  ```text
  ❌ Город "{input_name}" не найден
  ```

## 9. Матрица обработки ошибок

| Сценарий | Источник | Поведение скрипта | Пользовательский вывод | Код возврата |
|---|---|---|---|---|
| Город не найден геокодером | Open-Meteo Geocoding API | Пропустить город, продолжить | `❌ Город "X" не найден` | `≠ 0` в конце |
| Таймаут геокодирования | Сеть / API | Пропустить город, продолжить | `❌ Город "X": таймаут геокодирования` | `≠ 0` в конце |
| HTTP 5xx при геокодировании | Open-Meteo Geocoding API | Пропустить город, продолжить | `❌ Город "X": ошибка сервиса геокодирования` | `≠ 0` в конце |
| HTTP 4xx при геокодировании | Open-Meteo Geocoding API | Пропустить город, продолжить | `❌ Город "X": некорректный запрос к геокодеру` | `≠ 0` в конце |
| Ошибка декодирования JSON | Любой API | Пропустить город, продолжить | `❌ Город "X": некорректный ответ сервиса` | `≠ 0` в конце |
| Таймаут прогноза | Сеть / API | Пропустить город, продолжить | `❌ Город "X": таймаут прогноза погоды` | `≠ 0` в конце |
| HTTP 5xx при прогнозе | Open-Meteo Forecast API | Пропустить город, продолжить | `❌ Город "X": ошибка сервиса погоды` | `≠ 0` в конце |
| Превышен лимит городов | Валидация CLI | Завершить работу до обработки | `❌ Максимальное количество городов — 10` | `2` |
| Пустое или некорректное название | Валидация CLI | Пропустить аргумент | `⚠️ Пропущен пустой аргумент` | `≠ 0`, если не осталось городов |
| Успешная обработка всех городов | — | Стандартный вывод | Прогноз(ы) | `0` |

## 11. Нефункциональные требования

- Время выполнения для 1 города ≤ 5 секунд; для 10 городов ≤ 30 секунд.
- На каждый город не более 1 вызова геокодера и 1 вызова прогноза (с учётом кэша).
- Скрипт сохраняет работоспособность при отсутствии интернета для городов, находящихся в кэше.
- Все изменения не ломают существующий запуск без аргументов.

## 12. Критерии приёмки

1. `python weather_daily.py` выводит прогноз для Москвы.
2. `python weather_daily.py Москва "Санкт-Петербург"` выводит два отдельных сообщения в текущем формате.
3. `python weather_daily.py НесуществующийГород Москва` выводит ошибку по первому городу и прогноз для Москвы; код возврата не равен нулю.
4. `python weather_daily.py г1 г2 ... г11` выводит ошибку о превышении лимита в 10 городов.
5. Повторный запуск `python weather_daily.py Москва` не выполняет лишний HTTP-вызов к геокодеру (используется кэш).
6. Сообщения для иностранных городов (`Moscow`, `London`) формируются с учётом местного часового пояса.
7. Все изменения выполнены без добавления сторонних Python-пакетов.
