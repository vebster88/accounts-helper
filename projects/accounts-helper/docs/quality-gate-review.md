# Quality Gate Review — AccountsHelper

**Дата ревью:** 2026-07-22  
**Объект:** `docs/BRD.md`, `docs/HLD.md`, `docs/spec.md`  
**Версии:** BRD v1.0, HLD v1.0, Spec v1.0  
**Рецензент:** Hermes Agent / Quality gate sub-agent  
**Вердикт:** **CONDITIONALLY READY / УСЛОВНО ГОТОВ**  
**Оценка:** 82 / 100

## Краткое резюме

Спецификация AccountsHelper содержит полную декомпозицию BRD/HLD в функциональные и системные требования, модель данных, контракты сообщений, правила автоопределения полей, валидацию, обработку ошибок, тестовые сценарии и матрицу трассируемости. Документ структурирован, язык консистентен, критичных противоречий между BRD/HLD/Spec не выявлено. К вердикту CONDITIONALLY DOWNGRADE приводят 3 критических риска, требующих закрытия перед разработкой: отсутствие окончательного выбора KDF и параметров PBKDF2/Argon2id; неопределённость в использовании offscreen document для Argon2id в MV3; неутверждённая Privacy Policy для Chrome Web Store. Также есть ряд замечаний по согласованности идентификаторов, разделов HLD, безопасности 4-значного PIN и корректности матрицы трассируемости.

## Топ-5 находок

1. **Критический риск: KDF не выбран и не зафиксирован.** Spec/HLD декларируют PBKDF2 ≥600 000 итераций **или** Argon2id, но не фиксируют KDF по умолчанию, точные параметры и fallback. Это блокирует реализацию `crypto-service.ts` и замер NFR-02.  
2. **Критический риск: Argon2id в MV3 service worker.** Web Crypto API не содержит Argon2id; для реализации требуется offscreen document / WASM, что не отражено в Spec как обязательный путь и не согласовано.  
3. **Критический риск: Privacy Policy отсутствует.** Chrome Web Store требует Privacy Policy для расширений, обрабатывающих персональные/финансовые данные (REG-01); в Spec/HLD упомянута, но текст/место хранения не определены.  
4. **Мажорное замечание: 4-значный цифровой PIN + PBKDF2/Argon2id не обеспечивают высокую энтропию.** Ключевой пространство — 10 000 комбинаций; при KDF без rate-limiting offline-брутфорс возможен. В документах это признаётся (R-04), но не митигировано ограничением попыток на уровне хранилища (service-worker счётчик очищается при перезапуске).  
5. **Мажорное замечание: несоответствие идентификаторов сообщений в таблицах 6.2 и 6.3.** В таблице 6.3 присутствует `GET_FIELD_TYPE` с направлением SW → CS, тогда как в п.5.3 и п.7 алгоритм контекстного меню строится на основе `REQUEST_FIELD_TYPE` (CS → SW). Это создаёт путаницу в маршрутизации.

## Рекомендации по устранению

### Критические (блокируют переход к разработке)

1. **Выбрать и зафиксировать KDF и параметры.** Рекомендуется:
   - По умолчанию: **PBKDF2-SHA256 с 600 000 итераций** (реализуем в Web Crypto, NIST-approved).
   - Опционально: **Argon2id memory=64 MB, iterations=3, parallelism=1** через offscreen document/WASM, если Architect подтвердит совместимость с MV3.
   - Записать выбор в Spec 4.3 и HLD 6.2; добавить migration path между KDF при смене PIN.
2. **Определить реализацию Argon2id.** Если Argon2id остаётся в scope — добавить в HLD/Spec раздел об offscreen document (`chrome.offscreen`), загрузке WASM и fallback на PBKDF2. Иначе исключить Argon2id из MVP.
3. **Создать и приложить Privacy Policy.** Минимум: URL/файл, описывающий локальное хранение, отсутствие сетевых запросов, обработку кошельков/handle/телефона. Прописать в `manifest.json`.

### Мажорные

4. **Уточнить алгоритм checksum.** Указать: `checksum = HMAC-SHA256(key=KEK, message=ciphertext)` или иной конкретный механизм. Убедиться, что ключ HMAC не совпадает с DEK/KEK без дополнительного derivation, если это применимо.
5. **Унифицировать ключи chrome.storage.session.** Выбрать единый префикс `accountsHelper.*` (как в Spec 5.5) и обновить HLD 7.2.
6. **Уточнить схему сообщений автоопределения.** Удалить или переименовать дублирующий `GET_FIELD_TYPE` в таблице 6.3; добавить sequence-диаграмму: ПКМ → SW создаёт меню → CS определяет поле → SW обновляет меню.
7. **Добавить митигацию слабого PIN.** Запретить топ-20 простых PIN, добавить предупреждение об энтропии, рассмотреть rate-limiting на уровне service worker (например, экспоненциальная задержка между попытками).
8. **Привести BRD DoD / Spec Human Gate к статусу.** BRD status = `review`, Spec status = `draft`; перед разработкой рекомендуется привести оба документа к `approved` после закрытия критических рисков.

### Минорные

9. Добавить в Spec ссылку на конкретные разделы HLD (например, `[HLD-6.2]`).
10. Пометить BR-08 как `Should` в матрице Spec 12.1 и не понижать приоритет.
11. Уточнить пороговую логику при равенстве score двух типов (например, `wallet` + `email` одновременно).
12. Добавить AC для проверки 80% точности автоопределения на Google Forms / Typeform / кастомных формах.
13. Добавить TC для смены PIN (`CHANGE_PIN` в таблице 6.2 не покрыт тестом).
